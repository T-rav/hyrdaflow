# Architecture Knowledge System v1 — self-documenting HydraFlow

- **Status:** Draft
- **Date:** 2026-04-24
- **Author:** T-rav

## How to read this

This spec defines a self-documenting layer on top of HydraFlow that answers
two questions a human asks of a 1M-LOC autonomous system:

1. **What is this machine, conceptually?** — a curated *Functional Area Map*
   grouping loops, ports, and modules into ideas like *Orchestration*,
   *Caretaking*, *Quality Gates*, *Trust Fleet*, *Test Harness (MockWorld)*.
2. **What is true about it right now?** — eight generated artifacts
   (loop registry, port map, label state machine, module graph, event bus
   topology, ADR cross-reference matrix, MockWorld map, functional area
   index) regenerated from live source on a 4-hour tick by a new
   caretaker loop and CI-guarded against drift on every PR.

Both views are published as a **GitHub Pages** site (MkDocs Material) with
search, Mermaid diagrams, freshness badges, and a rolling changelog. The
existing wiki (`docs/wiki/`) and ADRs (`docs/adr/`) are rendered alongside
the generated artifacts and cross-linked.

The system is kept honest by **two writers, one written set**: a new
`DiagramLoop` (L24, joining the L9–L23 caretaker fleet) regenerates
artifacts autonomously and PRs the diff; the same code path runs in CI on
every PR in dry-run mode and fails the build if the working tree is stale.
The loop gives autonomous freshness; the CI guard gives same-PR catch.

Implementation lands in **three plans** under
`docs/superpowers/plans/2026-04-24-arch-knowledge-system-*.md`,
sequenced strictly: Plan A (extractors + 8 generators + runner),
Plan B (`docs/arch/` setup + `functional_areas.yml` + the
`functional_areas.py` generator that consumes it + ADR-0001 amendment),
Plan C (DiagramLoop + CI guard + Pages deploy + MkDocs). Sequencing and
dependencies in §13. When a plan disagrees with the spec, the spec wins and the plan
must be updated.

Hindsight has been removed from HydraFlow and is not represented in any
artifact. Residual test/audit references to it are out of scope for v1
and are flagged for a separate cleanup pass.

## 1. Context

HydraFlow already has substantial structured knowledge:

- **49 ADR files** under `docs/adr/` (0001–0049 with 0026 absent and 0043
  duplicated — historical artifacts), each with a mandatory `Enforced by:` line
  validated by `tests/test_adr_enforcement.py` (per `ADR-0044` P-discipline
  for decisions). The narrative layer is strong.
- **6 wiki files** under `docs/wiki/` (architecture, patterns, gotchas,
  testing, dependencies, index) totaling ~4,100 lines, partially maintained
  by the existing `RepoWikiLoop` (per `ADR-0032`).
- **12 hand-drawn `.likec4` files** under `docs/architecture/`, generated by
  past agent runs as one-shot snapshots from individual investigations.
- **A caretaker loop pattern** (`ADR-0029`) with ~14 named caretaker loops
  (L9–L23 in tests) that autonomously maintain various aspects of the
  system, plus a kill-switch convention (`ADR-0049`).
- **A hexagonal architecture** with named Ports (`PRPort`, `WorkspacePort`,
  `IssueStorePort`, `IssueFetcherPort`, `AgentPort`, `StateBackendPort`,
  `ReviewInsightStorePort`, `RouteBackCounterPort`) and a fake-adapter
  contract testing approach (`ADR-0047`).
- **MockWorld** (`tests/scenarios/fakes/mock_world.py`) as the foundation
  of the scenario test ring, per `ADR-0022`.

Three gaps justify this work:

1. **The likec4 diagrams have already drifted and will keep drifting.**
   They are agent-generated artifacts from individual past investigations,
   never updated when the underlying topology changes. `ADR-0001` says
   "five concurrent async loops" and the live system runs ~25. The
   diagrams *look* authoritative and are *actually* lying. There is no
   mechanism to detect the lie.
2. **The wiki and ADRs are markdown only — no rendered surface.** A
   human exploring the system reads raw `.md` files in GitHub's blob
   viewer, with no search, no nav tree, no Mermaid rendering, no
   cross-links beyond hand-authored relative paths. At 1M LOC this is the
   wrong primary surface for human consumption.
3. **There is no "what does this machine *do*" view.** Reading a list of
   25 loops doesn't tell you which are caretakers vs orchestration vs
   quality gates. The conceptual grouping lives only in tribal knowledge
   and scattered ADR prose.

**Stance.** This is not "more documentation." It is a *self-documenting
layer* — code that reads code and writes documentation, plus a publishing
surface that makes both the live truth and the human-curated narrative
visible. The same shape that has worked for `RepoWikiLoop` (an autonomous
loop maintaining a knowledge artifact) extends to architectural knowledge.

## 3. Stance — three layers, three decay rates

Treat documentation as a layered pipeline, not a folder. Each layer has a
different decay rate and a different update mechanism. **All three are
required**; none of them can substitute for any other.

| Layer | Decay rate | Owner | Examples in HydraFlow |
|---|---|---|---|
| **Generated** (live, no human author) | code speed | `DiagramLoop` + CI guard | `docs/arch/generated/*.md` (this spec) |
| **Curated** (semi-living, agent-maintained) | feature speed | `RepoWikiLoop` | `docs/wiki/*.md` |
| **Narrative** (slow, human-anchored) | decision speed | humans (with ADR-0040 reviewer assist) | `docs/adr/*.md` |

**Rule.** Each layer cites the next layer up via `module:symbol` or path
references; each layer is cited by the one below it. The Generated layer
cites Narrative (ADRs that justify the code); Narrative declares
`Enforced by:` references back to code; Curated explains *why* informally
and links to both. The published site materializes all three on equal
footing with cross-links.

**Corollary.** Hand-drawn architectural diagrams, when they exist, are
Curated-layer artifacts (story diagrams, motivation, rare events) and
must coexist with Generated equivalents under a drift-checked split. **In
v1 there are no curated diagrams** — the 12 existing `.likec4` files are
agent-generated, not human-curated, and are deleted as part of migration
(§8). The Functional Area Map (§4.3) is the only hand-curated content
in `docs/arch/` and it is YAML, not a diagram — its visual rendering is
generated.

## 4. Components

### 4.1 Extractors (`src/arch/extractors/`)

Pure, side-effect-free functions: source tree → typed Python model
objects. No I/O beyond reading files. Each extractor is unit-tested with
a fixture source tree.

| Module | What it produces |
|---|---|
| `loops.py` | List of `LoopInfo(name, module, tick_interval, event_subs, kill_switch_var, adr_refs, source_path)` — discovers all `BaseBackgroundLoop` subclasses via **AST static analysis** of `src/*.py` (parse → find `class X(BaseBackgroundLoop)`). Class introspection (`__subclasses__`) is **not** used because each loop module has deferred imports with side effects (config wiring, etc.); importing them all to enumerate would risk those side effects firing. AST is pure, side-effect-free, and matches how `tests/scenarios/catalog/loop_registrations.py` indirectly catalogs loops today. |
| `ports.py` | List of `PortInfo(name, module, methods, adapters: list[AdapterInfo], fake: FakeInfo \| None)` — discovers all `*Port` Protocols + their concrete and fake implementations. Per `ADR-0047`. |
| `labels.py` | `LabelStateMachine(states, transitions: list[(from, to, trigger)])` — extracts the label state machine from the transition code (per `ADR-0002`). |
| `modules.py` | `ModuleGraph(nodes, edges)` — package-level import graph for `src/`. |
| `events.py` | `EventBusTopology(events, publishers, subscribers)` — events published and subscribed via `EventBus`. |
| `adr_xref.py` | `ADRRefIndex(adr_to_modules, module_to_adrs)` — parses `module:symbol` references from each ADR's body and builds bidirectional mapping. |
| `mockworld.py` | `MockWorldMap(fakes: list[FakeInfo], scenario_uses: dict[str, list[str]])` — walks `tests/scenarios/fakes/` and indexes which scenarios wire each fake. |

**Determinism.** Each extractor sorts its outputs (alphabetical or
topological) so generator output is byte-stable for diff comparison.

### 4.2 Generators (`src/arch/generators/`)

Pure functions: in-memory model → Markdown string with embedded Mermaid.
Output is byte-stable (deterministic ordering, no timestamps in body —
timestamps live in `.meta.json` instead). Each generator emits a single
`.md` file under `docs/arch/generated/`.

**Naming.** To avoid extractor/generator module collisions when grepping,
the generator that consumes the `adr_xref` extractor is named
`adr_cross_reference.py` (the extractor stays as `adr_xref.py`). All other
generators share their extractor's stem since the inputs/outputs are
1:1 and the package separator (`extractors/` vs `generators/`) is
sufficient to disambiguate.

| Module | Output file | Content |
|---|---|---|
| `loop_registry.py` | `docs/arch/generated/loops.md` | Table of all loops with tick interval, event subs, kill-switch flag, ADR refs, source path. |
| `port_map.py` | `docs/arch/generated/ports.md` | Mermaid graph: Port → Adapter(s) → Fake; flagged rows for "Port without a fake" violations. |
| `label_state.py` | `docs/arch/generated/labels.md` | Mermaid `stateDiagram-v2` of all label transitions. |
| `module_graph.py` | `docs/arch/generated/modules.md` | Mermaid graph of package-level imports; cycles and cross-layer violations highlighted. |
| `event_bus.py` | `docs/arch/generated/events.md` | Bipartite Mermaid graph: events ↔ publishers/subscribers; orphan rows called out. |
| `adr_cross_reference.py` | `docs/arch/generated/adr_xref.md` | Two tables: ADR → modules cited; module → ADRs that cite it. |
| `mockworld_map.py` | `docs/arch/generated/mockworld.md` | Table of fakes with the Port each implements and the scenarios that wire them; Mermaid graph of fake → port → scenario coverage. |
| `functional_areas.py` | `docs/arch/generated/functional_areas.md` | The Functional Area Map page (§4.3). Reads `functional_areas.yml` + introspection from above; emits one Mermaid cluster per area + a per-area table linking to detail pages. |
| `changelog.py` | `docs/arch/generated/changelog.md` | Rolling 30/90-day arch-touching commit list (§6). |

### 4.4 DiagramLoop (L24)

A new caretaker loop joining the L9–L23 fleet.

- **Class.** `src/diagram_loop.py:DiagramLoop`, subclasses
  `BaseBackgroundLoop`.
- **Tick interval.** 4 hours (configurable via `HydraFlowConfig`).
  Cheap pure-Python operation; no network.
- **Kill-switch.** `HYDRAFLOW_DISABLE_DIAGRAM_LOOP=1` per `ADR-0049`
  convention. Enforced by `tests/test_diagram_loop_kill_switch.py`.
- **Tick behavior.**
  1. Run `python -m src.arch.runner --emit` against the current
     working tree → writes `docs/arch/generated/*.md` and updates
     `docs/arch/.meta.json`.
  2. `git status --porcelain docs/arch/generated docs/arch/.meta.json`.
  3. If empty: log "no drift" and exit. If non-empty: continue.
  4. Open (or update) a PR titled
     `chore(arch): regenerate architecture knowledge — YYYY-MM-DD`,
     body listing which artifacts changed and a 3-line summary per
     artifact (e.g., "loop_registry.md: +DiagramLoop, +1 loop").
  5. Auto-label `hydraflow-ready` so the standard auto-merger handles it.
- **Idempotence.** PR lookup uses **title-prefix match** (`chore(arch):
  regenerate architecture knowledge`) plus the `hydraflow-ready` label,
  not the full date-stamped title. This avoids a midnight-UTC race where
  a tick at 23:58 opens a PR with today's date and a tick at 00:05 would
  otherwise fail to find it under tomorrow's date and open a duplicate.
  When updating, the date in the title is refreshed to the current UTC
  date (existing pattern from `RepoWikiLoop`).
- **Coverage check.** As part of each tick, run
  `tests/architecture/test_functional_area_coverage.py`. If unassigned
  loops/ports are detected, open a separate
  `chore(arch): unassigned functional area` issue (not a silent failure).
- **Failure handling.** On extractor exception, log the traceback to
  `SessionLog`, do not open a PR, do not retry within the same tick;
  the next tick retries from scratch.

### 4.5 CI guard (`arch-regen.yml`)

Workflow runs on every PR. It does **not** write to the branch — its job
is to fail PRs whose `docs/arch/generated/` is stale relative to source.

```yaml
# .github/workflows/arch-regen.yml
name: arch-regen
on:
  pull_request:
    paths:
      - 'src/**'
      - 'docs/adr/**'
      - 'docs/arch/**'
      - 'tests/scenarios/fakes/**'
      - 'tests/architecture/**'
jobs:
  arch-check:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with: { python-version: '3.11' }
      - run: pip install -e '.[dev,test]'
      - run: python -m src.arch.runner --check
      - run: pytest tests/architecture -x
```

`--check` mode regenerates artifacts in a temp dir, diffs against
committed `docs/arch/generated/`, and exits 1 with a "run
`make arch-regen` and recommit" hint when the diff is non-empty.

```yaml

name: pages-deploy
on:
  push:
    branches: [main]
    paths:
      - 'docs/**'
      - 'mkdocs.yml'
      - 'src/arch/**'
      - 'pyproject.toml'

  contents: read
  pages: write
  id-token: write
concurrency:
  group: pages
  cancel-in-progress: false
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with: { fetch-depth: 0 }   # full history for changelog generator
      - uses: actions/setup-python@v5
        with: { python-version: '3.11' }
      - run: pip install -e '.[dev,docs]'
      - run: python -m src.arch.runner --emit  # idempotent regeneration
      - run: mkdocs build --strict
      - uses: actions/upload-pages-artifact@v3
        with: { path: ./site }
  deploy:
    needs: build
    runs-on: ubuntu-latest
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    steps:
      - id: deployment
        uses: actions/deploy-pages@v4
```

**Pages source.** Repo Settings → Pages → Source = "GitHub Actions"
(already enabled by user, 2026-04-24).

**No PR previews in v1.** Reviewers read changed `.md` files inline on
GitHub.

## 6. Freshness model

Every page in the site shows truthful staleness signals.

**Per-page footer (Generated layer only).**
> *Regenerated from commit `abc1234` on 2026-04-24 14:32 UTC. Source last
> changed at commit `def5678` (2026-04-24 12:08 UTC). Status: 🟢 fresh.*

**Badge states.**

| State | Trigger |
|---|---|
| 🟢 fresh | Regenerated within 24h **and** source unchanged since regen. |
| 🟡 source-moved | Source changed after last regen but within 7 days. (DiagramLoop should catch this within 4h; this state means the loop is paused or slow.) |
| 🔴 stale | Either >7 days since regen **or** an ADR-0001/ADR-0002 match test fails (i.e. the page contradicts an Accepted ADR) **or** `.meta.json` is absent / has no entry for this artifact (bootstrap case — first deploy before the loop has run). The "not yet generated" label appears in the tooltip. |

Implementation: `src/arch/freshness.py` reads `.meta.json` (per-artifact
SHA + regen-time) plus `git log -1 <source-paths>`; emits the badge as
the first line of each generated file's body. The MkDocs theme renders
emoji + tooltip.

**`/changelog/` page (rolling).**

`changelog.py` walks `git log --since=90.days.ago -- <pathspecs>` for
commits that touched `docs/arch/`, `docs/adr/`, `docs/wiki/`,
`src/arch/`, `mkdocs.yml`, or any of the source paths grouped by
functional area. Emits a chronological list grouped by week, with
subject lines and links to commits/PRs. Rebuilt on every site deploy.

**Performance.** At this repo's velocity (~30 PRs/day) a 90-day window is
~2,700 commits; `git log` with pathspecs runs in well under a second on
a full-history checkout. The Pages workflow already uses
`fetch-depth: 0` for this reason (§4.7).

**`/decisions/timeline/` page.**

ADRs sorted chronologically with a Mermaid graph drawing supersession
edges (Accepted → Superseded by 00XX). Generated by reading ADR
frontmatter and the `Related` section.

**"Why this exists" backlinks** (powered by §4.2 `adr_xref.py`).

Every Generated page links to the ADRs that cite the underlying source
(`adr_to_modules` reverse index). Every ADR rendered on the site shows
which Generated artifacts reference it (`module_to_adrs`). Every Wiki
entry that mentions a `module:symbol` becomes a backlink target.

## 7. Testing strategy (TDD per ADR-0044 P8/P10)

Every component lands red → green → refactor.

| Test file | What it asserts |
|---|---|
| `tests/architecture/test_extractor_loops.py` | `loops.py` against fixture source tree → expected `LoopInfo` list (sorted, deterministic). |
| `tests/architecture/test_extractor_ports.py` | `ports.py` against fixture source tree → expected `PortInfo` list. |
| `tests/architecture/test_extractor_labels.py` | `labels.py` against fixture transition code → expected state machine. |
| `tests/architecture/test_extractor_modules.py` | `modules.py` against fixture package tree → expected import graph. |
| `tests/architecture/test_extractor_events.py` | `events.py` against fixture EventBus uses → expected topology. |
| `tests/architecture/test_extractor_adr_xref.py` | `adr_xref.py` against fixture ADRs → expected bidirectional index. |
| `tests/architecture/test_extractor_mockworld.py` | `mockworld.py` against fixture `tests/scenarios/fakes/` → expected map. |
| `tests/architecture/test_generator_*.py` | Each generator: fixed model → byte-stable expected markdown. |
| `tests/architecture/test_curated_drift.py` | Run full pipeline against current `src/`, diff against committed `docs/arch/generated/`, fail on mismatch. **Invoked by `arch-regen.yml --check`.** |
| `tests/architecture/test_functional_area_coverage.py` | Every discovered loop and port appears in `functional_areas.yml`. Unassigned items fail with a list. |
| `tests/architecture/test_functional_areas_schema.py` | `functional_areas.yml` parses against a Pydantic schema; every required field is present per area. Lands in Plan B with the YAML it validates. |

## 10. Build & deploy

(Workflow YAML in §4.5 and §4.7.)

**Triggering matrix.**

| Event | `arch-regen.yml` | `pages-deploy.yml` | DiagramLoop |
|---|---|---|---|
| PR opened/updated | runs (CI guard) | does not run | does not directly trigger |
| Push to `main` | does not run | runs (deploy) | does not directly trigger |
| Loop tick (every 4h) | n/a | n/a | runs; opens PR if drift |
| Manual `gh workflow run` | available | available | n/a |

**Branch model.** No `gh-pages` branch. Pages serves the artifact
uploaded by `actions/deploy-pages@v4`. Single source of truth: `main`.

**Concurrency.** `pages-deploy.yml` uses
`concurrency: { group: pages, cancel-in-progress: false }` so multiple
merges queue rather than overwrite. DiagramLoop PRs are auto-merged via
`hydraflow-ready` like any other caretaker PR.

## 11. Risks & mitigations

| Risk | Mitigation |
|---|---|
| **PR noise from autonomous DiagramLoop** | Single combined PR per tick (not one per artifact); idempotent updates to existing open PR; auto-merge. Empirical baseline after 1 week informs whether 4h is the right interval. |
| **CI guard false positives on flaky generators** | Generators are pure functions over deterministic inputs — flakiness here is a bug, not noise. `tests/architecture/test_generator_*.py` enforces byte-stability. |
| **CI guard on DiagramLoop's own PR** | The guard runs on every PR, including DiagramLoop's regen PR — by construction the PR already contains the freshly regenerated state, so `--check` regenerates into a temp dir and finds an empty diff. Passes by design, provided generators are deterministic (covered by the row above). |
| **Malformed `functional_areas.yml` (human edit)** | A new `make arch-validate` target runs as part of `make quality`: it parses the YAML against a Pydantic schema and asserts every required field per area entry. A pre-commit Python check invokes the same target so syntax/schema errors never reach CI. The CI guard then catches semantic drift (unassigned new loops). Three-layer defense: pre-commit, `make quality`, CI. |
| **Functional Area Map drifts silently** | `test_functional_area_coverage.py` fails when a new loop/port is unassigned; DiagramLoop opens an issue, not a silent log line. |
| **ADR-0001 / ADR-0002 contradictions surface but no one fixes them** | The match tests fail in CI, blocking PRs that touch source until either the ADR is amended or the source is reconciled. The freshness badge goes 🔴 in the meantime. |
| **Mermaid rendering edge cases (e.g. cycles in module graph)** | `mkdocs build --strict` in CI catches render failures; `tests/architecture/test_mkdocs_strict.py` runs the same locally. |
| **`docs/arch/functional_areas.yml` gets out-of-sync with renamed classes** | Coverage test catches new unassigned items but not renames. Mitigation: add a `previous_names: [OldName]` field per area entry as a deprecation grace; flag rename collisions in the loop log. (YAGNI for v1; defer until a rename actually causes a problem.) |
| **GitHub Pages build minutes consumption** | Path-filtered triggers (only docs/src changes redeploy); the build itself is <2 minutes for a site this size. |
| **Hindsight residual references in tests look authoritative** | Out of scope for v1; flag a `hydraflow-find` issue for the cleanup pass. |
| **People hand-edit `docs/arch/generated/`** | A pre-commit hook (`hooks/pre-commit-arch.sh`) warns on staged changes under that path (not blocking — sometimes you genuinely want to edit during debugging). The DiagramLoop will overwrite the next tick; the warning is enough. |

## 13. Sequencing

Implementation lands in **three plans**, sequenced strictly (each plan's
landing unblocks the next).

**Plan A — runner foundation.**

- **Plan A — Extractors + Generators (8 of 9) + tests.** `src/arch/extractors/`,
  `src/arch/generators/` (excluding `functional_areas.py`),
  `src/arch/runner.py`, full unit-test suite per §7 except the
  functional-area coverage test. No DiagramLoop, no CI, no site yet.
  Ends with `make arch-regen` working locally and emitting 7
  generated artifacts (loops, ports, labels, modules, events, adr_xref,
  mockworld) plus the changelog stub.
- **Plan B — `docs/arch/` setup + `functional_areas.yml` +
  `functional_areas.py` generator + ADR-0001 amendment.** Hand-curation
  work; .likec4 deletion; CLAUDE.md update; YAML schema +
  `make arch-validate`; the `functional_areas.py` generator that
  consumes the YAML; `test_functional_area_coverage.py` and
  `test_functional_areas_schema.py`. **The generator and the YAML ship
  together** because the generator has nothing to read until the YAML
  exists. Plan B is sequenced *after* Plan A so the `runner.py` it
  hooks into already exists.

**Plan C — autonomy and publishing.**

- **DiagramLoop (L24) + CI guard + Pages deploy + MkDocs
  config + freshness model.** Wires Plan A's runner into both
  `arch-regen.yml` and `pages-deploy.yml`; adds the loop class and
  registers it (per §4.4); ships `mkdocs.yml` and the
  initial `docs/index.md` + `docs/about.md`. Ends with the published
  site live and the loop opening its first regen PR autonomously.

**Per-plan gating.** Each plan has its own
`superpowers:writing-plans` document, its own implementation cycle
(`superpowers:test-driven-development`), and its own review checkpoint
(`superpowers:requesting-code-review`). Plans land as separate PRs.

**Rollback.** Worst case for Plan C — disable the workflow (`gh
workflow disable pages-deploy`) and set
`HYDRAFLOW_DISABLE_DIAGRAM_LOOP=1` in the operator env. Plans A and B
are pure additions and have no rollback path beyond `git revert`.
